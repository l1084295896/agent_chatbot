### 2. 使用系统 Python 而非虚拟环境 Python
**问题描述**：终端输入 `python` 调用的是系统 Python（`C:/Python314`），导致 `ModuleNotFoundError: No module named 'langchain_chroma'`。

**原因**：Windows 环境变量 `PATH` 中系统 Python 排在 venv Python 前面。

**解决方案**：
1. 使用 venv 完整路径运行脚本：`g:/agent/venv/Scripts/python.exe g:/agent/rag/vector_store.py`
2. 在 VSCode `.vscode/settings.json` 中配置 `"python.defaultInterpreterPath": "g:/agent/venv/Scripts/python.exe"`
3. 创建 `run.bat` 脚本自动激活 venv 后运行

---

### 3. 命令行运行 Python 脚本报 `ModuleNotFoundError`
**问题描述**：
- `from utils.config_handler import chroma_conf` 报错，但 PyCharm 能正常运行
- 执行 `pip install -e .` 后，运行 `python g:/agent/agent/react_agent.py` 仍报错找不到 `agent` 模块

**原因**：Python 运行时不会自动将项目根目录加入 `sys.path`。PyCharm 会自动添加项目根目录到 `PYTHONPATH`，而标准 Python 运行脚本时只把脚本所在目录加入 `sys.path`。此外，如果 `__init__.py` 在 `pip install -e .` 执行之后才创建，包扫描不会包含这些目录。

**完整解决方案**：

1. **创建所有包的 `__init__.py` 文件**（空文件即可），使目录成为正式的 Python 包：
   ```bash
   touch utils/__init__.py
   touch agent/__init__.py
   touch agent/tools/__init__.py
   touch rag/__init__.py
   touch model/__init__.py
   ```

2. **确保项目根目录有 `pyproject.toml`**：
   ```toml
   [project]
   name = "agent-rag"
   version = "0.1.0"

   [tool.setuptools.packages.find]
   where = ["."]
   include = ["utils*", "model*", "rag*", "agent*"]
   ```

3. **执行可编辑模式安装**：
   ```bash
   pip install -e . --force-reinstall --no-deps
   ```

**验证安装成功**：
```bash
pip show agent-rag
python -c "import utils; import agent; import rag; print('All imports OK')"
```

---

### 4. `pip install -e .` 后出现 `agent_rag.egg-info` 文件夹
**问题描述**：项目目录中凭空出现 `agent_rag.egg-info` 文件夹，不确定是否可以删除。

**原因**：`pip install -e .`（可编辑模式安装）会由 setuptools 自动生成用于记录包的元数据。这是正常现象，不是错误。

**解决方案**：可以直接删除，不影响项目运行。

## 二、工具与配置

### 5. `get_abs_path` 函数路径问题
**问题描述**：`get_abs_path` 函数的路径是基于当前工作目录，不是全局

**解决方案**：修改为基于 `__file__` 所在目录计算绝对路径
```python
base_dir = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
return os.path.join(base_dir, relative_path)
```

### 7. Windows cmd 中文乱码
**问题描述**：cmd 使用系统默认编码（GBK），中文显示乱码

**解决方案**：
1. 按 Win + R 输入 `intl.cpl` 打开区域设置
2. 切换到"管理"选项卡
3. 点击"更改系统区域设置"

5. 点击确定并重启计算机

## 三、代码缺陷

**问题描述**：
- `.read` 漏了括号，只返回方法对象没有执行
- 错误日志信息写错（`[load_system_prompt]` 写成 `[load_rag_prompt]`）

**解决方案**：

- 修正错误日志信息

---

### 9. 向量库检索结果为 0 条
**问题描述**：检索词（如"扫地机器人"、"故障"）均无返回结果。

**原因**：`check_md5_hex` 函数调用逻辑错误，写成了 `if check_md5_hex:`（直接判断函数真假值），应该是 `if check_md5_hex(md5_hex):`（调用函数并传入参数）。导致所有文件均被判定为"已处理过"而跳过加载。

**解决方案**：修正为正确的函数调用写法：`if check_md5_hex(md5_hex):`。

---

### 10. 文本分割后序号不完整
**问题描述**：每段返回的字符串第一个序号缺失（如只有 `. 故障现象` 而非 `14. 故障现象`），且末尾仅有序号而无内容。

**原因**：
1. `chunk_size=200` 太小，同一个序号段落跨分割点时被截断
2. `separators` 列表中包含 `'.'`、`'?'`、`'!'` 等西文标点，会对中文序号（如 `14.`）进行误切割

**解决方案**：
1. 调大 `chunk_size` 至 400，`chunk_overlap` 调至 50
2. 调整 `separators` 列表，将 `""` 放在最前面作为兜底截断，中文标点靠后，西文标点移除
3. 删除 `chroma_db` 和 `md5.txt` 重新加载文档

---

### 11. PDF 加载后元数据包含大量不需要的字段
**问题描述**：PDF 的 `Document.metadata` 包含 `moddate`、`creator`、`producer`、`total_pages` 等大量元数据，只想知道 `source` 和 `page`。

**原因**：`PyPDFLoader` 默认将 PDF 的所有元数据全部写入 `Document.metadata`。

**解决方案**：在 `pdf_loader` 函数中加载后只保留需要的字段：
```python
doc.metadata = {"source": doc.metadata.get("source", file_path), "page": doc.metadata.get("page")}
```

---

### 12. 流式输出时用户提示词短暂出现在 assistant 对话框
**问题描述**：在 Streamlit 页面输入提示词并发送后，提示词不仅出现在 user 对话框中，还短暂出现在 agent 对话框的最前面，直到流式输出完成后才消失。

**原因**：`react_agent.py` 中 `execute_stream` 方法的 `last_content` 初始化为空字符串 `""`。模型首次返回的 chunk 内容包含用户的原始 query，由于 `new_content.startswith("")` 恒为 `True`，所以 delta 计算将整个 query 作为增量内容 yield 出来，导致用户输入的内容被当作 assistant 的输出先行显示。

**解决方案**：将 `last_content` 的初始值从 `""` 改为 `query`（用户输入本身）：
```python
last_content = query  # 而非 last_content = ""
```
